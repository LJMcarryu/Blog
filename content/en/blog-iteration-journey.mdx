## Phase 1: Code Auditing (Rounds 1-4)

### Typical Issues Found

After the initial build, I did a comprehensive audit and found 33 issues. Here are some representative ones:

**Empty dates causing NaN sort order:**

```typescript
// ❌ Before: empty dates produce NaN, undefined sort order
.sort((a, b) => new Date(b.date) - new Date(a.date))

// ✅ After: empty dates fall back to timestamp 0
.sort((a, b) => {
  const ta = a.date ? new Date(a.date).getTime() : 0;
  const tb = b.date ? new Date(b.date).getTime() : 0;
  return tb - ta;
});
```

**OG descriptions containing MDX markup:**

Social platform previews were showing raw `#`, `*`, and `import` statements in descriptions. Fixed by regex-stripping markdown syntax before truncating to 160 characters.

**Data hardcoded in components:**

Project lists, book data, and tool configurations were all inline in page components. Extracting them to a `data/` directory cleanly separated data from views.

### Audit Methodology

My audit checklist covered these dimensions:

- **Security**: XSS, external link `rel`, hardcoded secrets
- **TypeScript**: Strict mode, type assertions, dead code
- **React**: Key stability, Server/Client separation, useEffect cleanup
- **Accessibility**: ARIA labels, semantic HTML, keyboard navigation
- **SEO**: Metadata, sitemap, robots.txt, OG tags
- **i18n**: Message key consistency, locale coverage

After 4 rounds, the codebase reached **zero ESLint warnings, zero TypeScript errors**.

## Phase 2: Tiered Feature Development (Rounds 5-7)

With all bugs fixed, I organized desired features into three tiers by priority.

### Tier 1: Core Experience

These are "looks unprofessional without them" features:

- **Syntax highlighting**: `rehype-pretty-code` + Shiki with light/dark dual themes
- **RSS Feed**: RSS 2.0 format across all locales
- **Reading time**: Chinese at 400 chars/min, English at 200 words/min
- **Favicon**: Dynamically generated with `next/og`
- **Mobile navigation**: Hamburger menu at 640px breakpoint

### Tier 2: User Experience

These are "keeps users browsing longer" features:

- **Tag system**: Frontmatter `tags` field with URL parameter filtering
- **Table of Contents**: IntersectionObserver with active heading highlight
- **Code copy button**: Hover to reveal, "Copied!" feedback on click
- **Prev/Next navigation**: Bidirectional links at article bottom
- **Back to top**: Floating button after 400px scroll
- **Image lightbox**: Click to zoom, Escape to close
- **Site footer**: Social links + copyright

### Tier 3: Production Readiness

These are "the last mile before launch" features:

- **OG image generation**: Edge Runtime, 1200×630 dynamic images per post
- **JSON-LD**: Article schema on posts + WebSite schema on layout
- **Umami analytics**: Env-gated, zero output when unconfigured
- **Blog pagination**: 10 posts/page, compatible with search and tag filters
- **Unit tests**: Vitest covering core library functions, 17 tests passing

## Why Tiered Development Works

Breaking features into three tiers instead of doing everything at once has clear benefits:

1. **Every round builds successfully** — no "half-broken" intermediate states
2. **Clear priorities** — do what matters most first, not what's most fun
3. **Always deployable** — you can ship at any point

## Takeaway

Many people think building a blog is a one-time thing — build it and you're done. In reality, **building is 20%, iterating is 80%**.

Code auditing taught me to systematically review my own work. Feature tiering taught me to manage complexity. And the whole process? It's the best engineering practice you can get.
