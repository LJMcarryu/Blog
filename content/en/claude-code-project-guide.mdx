## Phase 1: Project Planning — From Requirements to CLAUDE.md

### 1.1 Requirements Analysis with Claude Code

When you have a vague idea, the first step is not writing code — it is letting Claude help you think it through.

```text
You: I want to build a CLI tool that checks links in Markdown files.
     The idea: give it a directory, it scans all .md files, extracts links,
     checks each one, and outputs a report. What do you think?

Claude: This is a very practical tool. Let me break it down:

     Core modules:
     1. File scanner - glob pattern matching for .md/.mdx files
     2. Link extractor - regex parsing of [text](url) and bare links
     3. Link checker - HTTP HEAD request validation with concurrency
     4. Report generator - colored terminal output + optional JSON

     Tech stack suggestion:
     - TypeScript + Node.js (type safety, rich ecosystem)
     - commander.js (CLI argument parsing)
     - chalk (colored terminal output)
     - glob (file matching)

     Key considerations:
     - Concurrency control: cannot fire hundreds of HTTP requests at once
     - Timeout handling: some servers respond slowly
     - Relative vs absolute links: different handling logic
     - Anchor links: #section needs heading existence check
```

Notice the prompt strategy here: **describe the idea first, let Claude analyze feasibility and architecture**, rather than jumping straight to "write me a link checker." This surfaces potential issues before you start coding.

## Phase 2: Core Development — Prompt-Driven Implementation

### 2.1 Step-by-Step Development Strategy

A common mistake when developing multi-module projects is asking Claude to write everything at once. A better strategy: **give the big picture first, then implement module by module**.

```text
You: We will develop in this order:
     1. scanner (file scanning) — simplest, no external dependencies
     2. extractor (link extraction) — pure string processing
     3. checker (link checking) — most complex, involves network requests
     4. reporter + CLI entry — wire everything together

     Run tests after each module. Start with scanner.
```

Benefits of this approach:
- Claude knows the overall architecture, so interface designs stay consistent
- Incremental implementation, each step verifiable, easy to pinpoint issues
- Context window does not explode all at once

> This is the "zoom rhythm" from the [Prompt Guide](/en/blog/claude-code-prompt-guide) — zoom out for the big picture, then zoom in for details.

### 2.2 File Scanner Module: scanFiles()

```text
You: Implement src/scanner.ts:
     - Takes a directory path and optional glob pattern (default **/*.md)
     - Returns array of matching file paths
     - Handle non-existent directory
```

```typescript
// src/scanner.ts
import { glob } from "glob";
import { access } from "fs/promises";

export interface ScanOptions {
  /** glob 模式，默认匹配所有 .md 文件 */
  pattern?: string;
  /** 要忽略的目录 */
  ignore?: string[];
}

const DEFAULT_PATTERN = "**/*.md";
const DEFAULT_IGNORE = ["node_modules/**", ".git/**"];

/**
 * 扫描目录中的 Markdown 文件
 */
export async function scanFiles(
  dir: string,
  options: ScanOptions = {}
): Promise<string[]> {
  const { pattern = DEFAULT_PATTERN, ignore = DEFAULT_IGNORE } = options;

  try {
    await access(dir);
  } catch {
    throw new Error("Directory not found: " + dir);
  }

  const files = await glob(pattern, {
    cwd: dir,
    absolute: true,
    ignore,
  });

  return files.sort();
}
```

Clean, type-safe, proper error handling. This is the output quality you get when you give Claude clear constraints.

### 2.3 Link Extractor Module: extractLinks()

This module uses the "read first, then modify" pattern — let Claude understand the various Markdown link formats before writing extraction logic.

```text
You: Implement src/extractor.ts. Markdown links come in these formats:
     1. [text](url) — standard links
     2. [text](url "title") — links with titles
     3. ![alt](url) — images
     4. <https://example.com> — autolinks
     5. [text][ref] + [ref]: url — reference links

     Handle the first 4 for now, add reference links later.
     Each link should record: url, line number, raw text.
```

```typescript
// src/extractor.ts
import { readFile } from "fs/promises";

export interface LinkInfo {
  /** 链接 URL */
  url: string;
  /** 所在行号（从 1 开始） */
  line: number;
  /** 原始匹配文本 */
  raw: string;
  /** 来源文件路径 */
  file: string;
}

// 匹配 [text](url) 和 ![alt](url)
const INLINE_LINK = /!?\[([^\]]*)\]\(([^)\s]+)(?:\s+"[^"]*")?\)/g;
// 匹配 <https://...>
const AUTO_LINK = /<(https?:\/\/[^>]+)>/g;

/**
 * 从 Markdown 文件中提取所有链接
 */
export async function extractLinks(filePath: string): Promise<LinkInfo[]> {
  const content = await readFile(filePath, "utf-8");
  const lines = content.split("\n");
  const links: LinkInfo[] = [];

  for (let i = 0; i < lines.length; i++) {
    const line = lines[i];
    const lineNum = i + 1;

    // 跳过代码块
    if (line.trimStart().startsWith("```")) continue;

    // 标准链接和图片
    let match: RegExpExecArray | null;
    INLINE_LINK.lastIndex = 0;
    while ((match = INLINE_LINK.exec(line)) !== null) {
      links.push({
        url: match[2],
        line: lineNum,
        raw: match[0],
        file: filePath,
      });
    }

    // 自动链接
    AUTO_LINK.lastIndex = 0;
    while ((match = AUTO_LINK.exec(line)) !== null) {
      links.push({
        url: match[1],
        line: lineNum,
        raw: match[0],
        file: filePath,
      });
    }
  }

  return links;
}
```

> Note the code block skipping logic is still naive (no multi-line code block state tracking). This is something we will improve in the refactoring phase.

### 2.4 Link Checker Module: checkLink()

The most complex module, involving network requests and concurrency.

```text
You: Implement src/checker.ts:
     - HTTP/HTTPS links: send HEAD request, 10s timeout
     - Relative links: check if file exists
     - Skip mailto: and tel: links
     - Return check result (ok/broken/skipped + reason)
     - No concurrency control for now, we will add it in the debugging phase
```

```typescript
// src/checker.ts
import { access } from "fs/promises";
import { resolve, dirname } from "path";
import { LinkInfo } from "./extractor.js";

export type LinkStatus = "ok" | "broken" | "skipped";

export interface CheckResult {
  link: LinkInfo;
  status: LinkStatus;
  statusCode?: number;
  error?: string;
}

const SKIP_PROTOCOLS = ["mailto:", "tel:", "javascript:"];
const TIMEOUT_MS = 10_000;

/**
 * 检查单个链接的可访问性
 */
export async function checkLink(link: LinkInfo): Promise<CheckResult> {
  const { url } = link;

  // 跳过特殊协议
  if (SKIP_PROTOCOLS.some((p) => url.startsWith(p))) {
    return { link, status: "skipped", error: "Special protocol" };
  }

  // 锚点链接暂时跳过
  if (url.startsWith("#")) {
    return { link, status: "skipped", error: "Anchor link" };
  }

  // HTTP/HTTPS 链接
  if (url.startsWith("http://") || url.startsWith("https://")) {
    return checkHttpLink(link);
  }

  // 相对链接
  return checkRelativeLink(link);
}

async function checkHttpLink(link: LinkInfo): Promise<CheckResult> {
  try {
    const controller = new AbortController();
    const timeout = setTimeout(() => controller.abort(), TIMEOUT_MS);

    const response = await fetch(link.url, {
      method: "HEAD",
      signal: controller.signal,
      headers: { "User-Agent": "mdcheck/0.1.0" },
      redirect: "follow",
    });

    clearTimeout(timeout);

    if (response.ok) {
      return { link, status: "ok", statusCode: response.status };
    }

    // 有些服务器不支持 HEAD，降级到 GET
    if (response.status === 405) {
      const getResponse = await fetch(link.url, {
        method: "GET",
        signal: AbortSignal.timeout(TIMEOUT_MS),
        headers: { "User-Agent": "mdcheck/0.1.0" },
        redirect: "follow",
      });
      return {
        link,
        status: getResponse.ok ? "ok" : "broken",
        statusCode: getResponse.status,
      };
    }

    return { link, status: "broken", statusCode: response.status };
  } catch (err) {
    const message = err instanceof Error ? err.message : String(err);
    return { link, status: "broken", error: message };
  }
}

async function checkRelativeLink(link: LinkInfo): Promise<CheckResult> {
  const targetPath = resolve(dirname(link.file), link.url.split("#")[0]);
  try {
    await access(targetPath);
    return { link, status: "ok" };
  } catch {
    return { link, status: "broken", error: "File not found" };
  }
}
```

A few notable design decisions:
- HEAD-to-GET fallback when servers reject HEAD requests
- AbortController for timeout control
- Relative links strip anchors before checking file existence

Deliberately no concurrency control — this will surface as a bug in Phase 4.

```typescript
// src/reporter.ts
import chalk from "chalk";
import { CheckResult } from "./checker.js";

export function reportToConsole(results: CheckResult[]): void {
  const broken = results.filter((r) => r.status === "broken");
  const ok = results.filter((r) => r.status === "ok");
  const skipped = results.filter((r) => r.status === "skipped");

  if (broken.length > 0) {
    console.log(chalk.red.bold("\n✗ " + broken.length + " broken link(s) found:\n"));
    for (const r of broken) {
      console.log(chalk.red("  ✗ " + r.link.url));
      console.log(chalk.gray("    " + r.link.file + ":" + r.link.line));
      if (r.statusCode) console.log(chalk.gray("    HTTP " + r.statusCode));
      if (r.error) console.log(chalk.gray("    " + r.error));
      console.log();
    }
  }

  console.log(chalk.bold("Summary:"));
  console.log(chalk.green("  ✓ " + ok.length + " ok"));
  console.log(chalk.red("  ✗ " + broken.length + " broken"));
  console.log(chalk.yellow("  ⊘ " + skipped.length + " skipped"));
}

export function reportToJson(results: CheckResult[]): void {
  const output = results.map((r) => ({
    url: r.link.url,
    file: r.link.file,
    line: r.link.line,
    status: r.status,
    statusCode: r.statusCode,
    error: r.error,
  }));
  console.log(JSON.stringify(output, null, 2));
}
```

```typescript
// src/index.ts
#!/usr/bin/env node
import { program } from "commander";
import { scanFiles } from "./scanner.js";
import { extractLinks } from "./extractor.js";
import { checkLink } from "./checker.js";
import { reportToConsole, reportToJson } from "./reporter.js";

program
  .name("mdcheck")
  .description("Check Markdown files for broken links")
  .version("0.1.0")
  .argument("<dir>", "Directory to scan")
  .option("-p, --pattern <glob>", "File glob pattern", "**/*.md")
  .option("-i, --ignore <dirs...>", "Directories to ignore")
  .option("--json", "Output results as JSON")
  .action(async (dir, options) => {
    try {
      const files = await scanFiles(dir, {
        pattern: options.pattern,
        ignore: options.ignore?.map((d: string) => d + "/**"),
      });

      if (files.length === 0) {
        console.log("No Markdown files found.");
        process.exit(0);
      }

      console.log("Found " + files.length + " file(s), extracting links...");

      const allLinks = (await Promise.all(files.map(extractLinks))).flat();
      console.log("Found " + allLinks.length + " link(s), checking...");

      // 注意：这里没有并发控制，所有链接同时检查
      const results = await Promise.all(allLinks.map(checkLink));

      if (options.json) {
        reportToJson(results);
      } else {
        reportToConsole(results);
      }

      const hasBroken = results.some((r) => r.status === "broken");
      process.exit(hasBroken ? 1 : 0);
    } catch (err) {
      console.error(err instanceof Error ? err.message : err);
      process.exit(2);
    }
  });

program.parse();
```

Notice `Promise.all(allLinks.map(checkLink))` — all links checked simultaneously with no concurrency control. Works fine with few links, but breaks with many. We intentionally leave this problem for Phase 4.

### 3.3 Tests for checkLink

Network request tests need mocking. Here we use `msw` (Mock Service Worker) to intercept HTTP requests:

```text
You: Write tests for checkLink. Mock HTTP requests, no real requests.
     Cover:
     1. Normal 200 response → ok
     2. 404 response → broken
     3. Timeout → broken
     4. mailto link → skipped
     5. Relative link to existing file → ok
     6. Relative link to missing file → broken
```

```typescript
// tests/checker.test.ts
import { describe, it, expect, beforeAll, afterAll, beforeEach, afterEach } from "vitest";
import { http, HttpResponse } from "msw";
import { setupServer } from "msw/node";
import { checkLink, CheckResult } from "../src/checker.js";
import { LinkInfo } from "../src/extractor.js";
import { writeFile, mkdir, rm } from "fs/promises";
import { join } from "path";

const TEST_DIR = join(__dirname, ".tmp-checker-test");

// Mock HTTP server
const server = setupServer(
  http.head("https://example.com/ok", () => {
    return new HttpResponse(null, { status: 200 });
  }),
  http.head("https://example.com/not-found", () => {
    return new HttpResponse(null, { status: 404 });
  }),
  http.head("https://example.com/slow", async () => {
    await new Promise((r) => setTimeout(r, 15000));
    return new HttpResponse(null, { status: 200 });
  })
);

beforeAll(() => server.listen());
afterAll(() => server.close());

beforeEach(async () => {
  await mkdir(TEST_DIR, { recursive: true });
  server.resetHandlers();
});

afterEach(async () => {
  await rm(TEST_DIR, { recursive: true, force: true });
});

function makeLink(url: string, file?: string): LinkInfo {
  return { url, line: 1, raw: url, file: file || join(TEST_DIR, "test.md") };
}

describe("checkLink", () => {
  it("should return ok for 200 response", async () => {
    const result = await checkLink(makeLink("https://example.com/ok"));
    expect(result.status).toBe("ok");
    expect(result.statusCode).toBe(200);
  });

  it("should return broken for 404 response", async () => {
    const result = await checkLink(makeLink("https://example.com/not-found"));
    expect(result.status).toBe("broken");
    expect(result.statusCode).toBe(404);
  });

  it("should return broken on timeout", async () => {
    const result = await checkLink(makeLink("https://example.com/slow"));
    expect(result.status).toBe("broken");
    expect(result.error).toBeDefined();
  });

  it("should skip mailto links", async () => {
    const result = await checkLink(makeLink("mailto:test@example.com"));
    expect(result.status).toBe("skipped");
  });

  it("should return ok for existing relative file", async () => {
    const targetFile = join(TEST_DIR, "readme.md");
    await writeFile(targetFile, "hello");
    const result = await checkLink(makeLink("readme.md"));
    expect(result.status).toBe("ok");
  });

  it("should return broken for missing relative file", async () => {
    const result = await checkLink(makeLink("nonexistent.md"));
    expect(result.status).toBe("broken");
  });
});

## Phase 4: Debugging in Action — Solving a Real Bug

### 4.1 The Bug Scenario

At this point, basic functionality is complete. You excitedly test it on a real documentation directory:

```bash
$ npx tsx src/index.ts ./docs

Found 23 file(s), extracting links...
Found 187 link(s), checking...
Error: fetch failed
    at Object.fetch (node:internal/deps/undici/undici:11576:11)
    cause: Error: connect EMFILE 93.184.216.34:443
        at TCPConnectWrap.afterConnect [as oncomplete] (node:net:1595:16)
```

187 links all firing HTTP requests simultaneously — the operating system runs out of file descriptors (sockets).

A classic concurrency problem: works perfectly with few links, crashes with many.

### 4.2 Root Cause Analysis with Claude Code

```text
You: Running mdcheck on a directory with 187 links throws this error:
     Error: fetch failed
     cause: Error: connect EMFILE 93.184.216.34:443

     Full stack trace above. Help me analyze the cause.

Claude: This is a classic concurrent resource exhaustion problem.

     Root cause analysis:
     1. In src/index.ts line 34, Promise.all(allLinks.map(checkLink))
        fires 187 HTTP requests simultaneously
     2. Each HTTP request needs a TCP socket (file descriptor)
     3. The OS limits file descriptors per process (typically 1024)
     4. While 187 connections is under 1024, DNS resolution and
        TLS handshake queuing actually consume more descriptors
     5. EMFILE error means "too many open files"

     Solution:
     Use p-limit for concurrency control, limiting simultaneous
     HTTP requests. Recommended concurrency: 10-20.
```

Notice the prompt strategy: **paste the error directly** and let Claude analyze. No need to guess the cause yourself. Claude can determine root cause from the error stack, error code, and surrounding code context.

export async function checkLinks(

### 6.4 Hooks for Safety

Configure Claude Code Hooks to automatically run tests before commits:

```json
// .claude/settings.json（添加 hooks）
{
  "permissions": {
    "allow": [
      "Bash(npm test)",
      "Bash(npm run build)",
      "Bash(npm run lint)"
    ]
  },
  "hooks": {
    "PreCommit": [
      {
        "matcher": "**/*.ts",
        "command": "npm test"
      }
    ]
  }
}
```

Now every time Claude tries to commit code, tests run first. If tests fail, the commit is blocked — preventing bad code from entering the repository at the source.

## Lessons Learned: Best Practices for AI-Assisted Development

### 8.1 Project-Level Prompt Strategy

Across the entire project, prompts follow a clear "zoom rhythm":

- **Planning phase**: Zoom out — describe overall requirements, let Claude analyze architecture
- **Development phase**: Zoom in — focus on individual modules, give precise interface constraints
- **Review phase**: Zoom out — let Claude review overall code quality

This rhythm is not linear — it repeats within each phase too. For example, when developing the checker module, we zoomed out to discuss concurrency strategy, then zoomed in to write the implementation.

### 8.2 Session Management Strategy

A complete project should not be built in a single session. Recommended approach:

- **Use `/clear` between phases**: After completing each phase, clear context, start fresh
- **CLAUDE.md is cross-session memory**: Project conventions live in CLAUDE.md, readable by every new session
- **Git commits are checkpoints**: Commit at the end of each phase, start the next session from a clean state

```text
# Typical session rhythm
Session 1: Planning + init + CLAUDE.md → git commit → /clear
Session 2: scanner + extractor modules → git commit → /clear
Session 3: checker + reporter + CLI → git commit → /clear
Session 4: Write tests + run coverage → git commit → /clear
Session 5: Debug + fix → git commit → /clear
Session 6: Refactor + CI + finishing → git commit
```
