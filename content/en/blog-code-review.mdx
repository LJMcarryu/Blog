## Why Do a Code Review?

The blog has been running for a while and everything works. But "it works" and "it works well" are two different things. Before technical debt piles up, I decided to do a thorough audit across six dimensions: **performance, security, accessibility, SEO, code quality, and i18n**.

---

## Performance Issues

### Oversized Video Background

The homepage background videos are the biggest bottleneck:

- `home_bg_video_dark.mp4` — 20.7 MB
- `home_bg_video_light.mp4` — 10.8 MB

Over 31 MB total. Brutal on mobile. The fix:

1. Compress and convert to WebM/VP9 with FFmpeg — 60-70% size reduction
2. Add a `poster` attribute to `<video>` so the first frame loads as a static image
3. Consider replacing video with a static image on mobile

### Fragile Timers in Components

`CodeCopyButton.tsx` uses `setTimeout(100ms)` to wait for DOM rendering before inserting copy buttons. This can fail on slower devices. A `MutationObserver` would be more reliable.

`TableOfContents.tsx` has a similar 150ms initialization delay that could cause CLS (Cumulative Layout Shift).

### Video Reload on Theme Switch

`ThemedVideo.tsx` calls `.load()` when the theme changes, causing a visible flash. A better approach: render both `<video>` elements and toggle visibility with CSS `display`.

## Security Issues

### Unsanitized OG Image Parameters

`app/og/route.tsx` reads `title` from URL params and renders it directly into `ImageResponse` without length limits or content validation. While the OG image generation environment is relatively safe, it's still an external input entry point worth sanitizing.

### Silent MDX Parsing Failures

`lib/posts.ts` swallows errors in its catch block — no logging, no re-throw:

```typescript
// Current approach — completely silent on error
catch {
  return { slug, title: "", date: "", description: "", tags: [], content: "" };
}
```

At minimum, add a `console.warn`. Otherwise, a broken MDX file is invisible.

## Accessibility Issues

### Lightbox Missing Focus Management

`Lightbox.tsx` simulates a modal with a `div` but doesn't implement a focus trap. Keyboard users can Tab out of the lightbox and interact with background elements.

Recommended fix: use the native `<dialog>` element, or add `focus-trap-react`.

### Inconsistent Copy Button aria-label

`CodeCopyButton.tsx` shows "Copy" as button text but has `aria-label="Copy code"` — screen readers and visual presentation don't match. Pick one and stick with it.

## SEO Issues

### Metadata Description Not Locale-Aware

`layout.tsx` sets the metadata description to `"个人博客 | Personal Blog"` — mixing both languages. Search engines match description language to page language, so this should be set dynamically per locale.

### Invalid RSS Language Tag

`feed.xml/route.ts` sets the RSS `language` field to `"zh-en"`, which is not a valid RFC 5646 language tag. Either generate separate feeds per language, or use `"zh"` as the primary.

### Sitemap Missing Change Frequency

`sitemap.ts` doesn't include `changeFrequency` or `priority` fields. While Google claims to mostly ignore these, other search engines (like Bing) still reference them.

### Dangerous robots.ts Fallback

Both `robots.ts` and `sitemap.ts` fall back to `https://example.com` when `NEXT_PUBLIC_SITE_URL` is not set. If you forget to configure this in production, your sitemap and robots.txt will point to the wrong domain — an SEO disaster.

## Code Quality

### `!important` Overuse in CSS

`globals.css` is littered with `!important`:

- 20 `nth-child` animation rules (lines 90-109)
- Code block backgrounds (lines 223, 227)
- Shiki theme colors (lines 252-256)
- Inline code styles (lines 317-324)

The more `!important` you have, the harder maintenance gets. Consider restructuring selector specificity or using CSS Layers.

### Hardcoded Color Values

Code block text colors are hardcoded as `#0a0a0a` and `#f0f0f0` instead of using CSS variables. Inconsistent with the `var(--fg)` pattern used everywhere else.

### Redundant Pagination Logic

`blog/page.tsx` uses `Math.max(1, Math.ceil(...))` — redundant since `Math.ceil()` already returns ≥ 1 for positive numbers.

### Hardcoded Data in About Page

`about/page.tsx` has a SKILLS array defined inline, while projects and books are properly extracted to `data/`. Move it to `data/skills.ts` for consistency.

## i18n Issues

### Hardcoded Strings Bypassing Translation

Several places use hardcoded Chinese/English strings instead of `next-intl` translations:

- `blog/[slug]/page.tsx` line 182 — prev/next post labels
- `blog/page.tsx` line 85 — "全部" / "All" tag filter

These should be in `messages/zh.json` and `messages/en.json`.

### Fragile Type Assertion

`i18n/request.ts` uses `as "zh" | "en"` for type assertion. If a new locale is added (say Japanese), this silently breaks. Derive the type from `routing.locales` instead.

## Second Review: Deeper Issues

After completing the first five rounds of optimization, I did another deep review and found some subtler problems.

### Incomplete CSS Variable Coverage

Although `--border` and `--bg-subtle` design tokens were introduced earlier, `globals.css` still had 15+ raw `rgba(125, 125, 125, ...)` values scattered across scrollbar styles, code block borders, inline code, TOC, blockquotes, and more. Same semantic values, not using the variable system — changing one means hunting down a dozen places.

### Giscus Initialization Theme Bug

The initialization effect in `Comments.tsx` depended on `[locale, theme]`, but `theme` starts as `"light"` (React state default), not the actual page theme. In dark mode, Giscus would first render with a light theme, then flash to dark when the postMessage effect fires.

### Accessibility Gaps

- Hamburger menu button missing `aria-controls` — screen readers can't associate the button with the menu
- Active tag filter only has visual styling (`data-active`), no `aria-current` semantic marker
- Lightbox `alt` text hardcoded to `"Enlarged preview"`, losing the original image description
- Search input's `focus:ring-2` has no color specified — focus ring invisible in some browsers

### Redundant Code

- `blog/[slug]/page.tsx` sets both `dynamicParams = false` and `dynamic = 'force-static'` — the latter is redundant
- `Footer.tsx` and homepage set `target="_self"` for mailto links — that's the browser default
- RSS feed description mixes languages: `"个人博客 | Personal Blog"` — not RFC-compliant

## Third Review: Missed Spots and Redundancy

After the sixth round of optimization, another review uncovered several overlooked issues.

### Design Token Gap

`SearchBar.tsx` still had a raw `rgba(125, 125, 125, 0.05)` — the search input background was missed during the design token migration.

### Pagination URL Anomaly

`blog/page.tsx` returned `"?"` as `baseHref` when no filter params were active, producing URLs like `/blog?`. Functionally harmless, but the URL isn't clean.

### Giscus Language Switch Broken

`Comments.tsx` used an `initialized` ref to prevent duplicate initialization, but this also blocked re-initialization on locale change. After switching languages, the comment section's UI language wouldn't update.

### Redundant CSS Variable Declarations

The `.dark` block contained 11 CSS variables identical to `:root` (all rgba-based tokens, `--star`, `--success`). These values don't change between modes — the duplication only adds maintenance burden.

### RSS Description Language Mismatch

The RSS feed's `<description>` was in Chinese only, but the feed actually includes posts in both languages. The description should be more universal.

## Fourth Review: Architecture and i18n

### Duplicate Dark Mode Observer Logic

`ThemedVideo.tsx`, `Comments.tsx`, and `ThemeToggle.tsx` each create their own MutationObserver on `document.documentElement` class changes. The first two have identical logic — should be extracted to a shared `useDarkMode` hook.

### UI Component Labels Hardcoded in English

Four client components have hardcoded English aria-labels and button text, bypassing i18n:

- `ThemeToggle` — "Switch to light/dark mode"
- `BackToTop` — "Back to top"
- `CodeCopyButton` — "Copy" / "Copied!" / "Failed" / "Copy code"
- `LanguageSwitcher` — "Toggle language"

Chinese users see English interaction prompts — inconsistent experience.

### Missing Canonical and Alternate URLs

Neither the layout nor blog post pages set `alternates` metadata. Search engines may treat `/zh/blog/xxx` and `/en/blog/xxx` as duplicate content, hurting SEO.

### TableOfContents Unnecessary Rebuilds

The MutationObserver re-initializes the IntersectionObserver on every DOM change, including when CodeCopyButton inserts buttons. Should cache the heading list and only rebuild when headings actually change.

---

## Fifth Review: i18n Blind Spots

### Reading Time Hardcoded

`lib/reading-time.ts` returns pre-formatted strings like `"X 分钟阅读"` / `"X min read"`, bypassing the translation system. Utility functions shouldn't contain UI copy.

### Book Notes Chinese-Only

`data/books.ts` book review notes are plain Chinese strings — English users see Chinese content. Should use `Record<"zh" | "en", string>` like `projects.ts` does.

### Stars Component aria-label Hardcoded

The star rating `aria-label` in `books/page.tsx` is hardcoded as `"X stars"` — Chinese screen reader users hear English.

### Photo Alt Text Fallback Broken

`photos/page.tsx`'s `altFromPath()` uses `??` for the fallback, but `??` doesn't catch empty strings. If a filename is just an extension (e.g., `.jpg`), alt text would be empty.

## Takeaway

Overall, the project is well-structured. The Next.js App Router + next-intl combo is solid, and the dark mode and animation systems are nicely done. After five rounds of review, the main optimization opportunities fall into five buckets:

1. **Performance**: Video compression is the single biggest win
2. **Robustness**: Error handling and environment variable validation need attention
3. **Consistency**: The CSS variable system needs to be thorough — no raw values left behind
4. **Accessibility**: aria attributes, focus management, semantic markers, and UI label i18n need to be filled in one by one
5. **SEO**: canonical/alternate URLs and structured data need completion

None of these are urgent, but fixing them incrementally will make the project significantly more maintainable. Next step: tackle them by priority.
