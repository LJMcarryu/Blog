## Background

After completing the blog's feature development, I noticed a warning during production builds: blog post pages were marked as "dynamic rendering." This meant every user visit required server-side rendering instead of serving pre-generated static HTML.

For a content-focused blog, this is wasteful in both performance and cost. This article documents how I solved this issue and fixed several other critical bugs along the way.

## Issue 1: Blog Posts Rendered Dynamically

### Symptoms

Running `npm run build` showed a warning:

```
The path /en/blog/blog-iteration-journey is marked as "dynamic"
since it will be rendered for each user at request time.
```

### Root Cause

Although I had implemented `generateStaticParams()`, Next.js still chose dynamic rendering. Common causes include:

1. Using dynamic APIs (`cookies()`, `headers()`, `searchParams`)
2. Using `cache: 'no-store'` in fetch calls
3. Not explicitly telling Next.js to force static generation

### Solution

Added two configurations in `app/[locale]/blog/[slug]/page.tsx`:

```typescript
export async function generateStaticParams() {
  const locales = ["zh", "en"];
  const params: { locale: string; slug: string }[] = [];

  for (const locale of locales) {
    const posts = getPostsByLocale(locale);
    posts.forEach((post) => {
      params.push({ locale, slug: post.slug });
    });
  }

  return params;
}

// ✅ New: Force static generation
export const dynamicParams = false; // 404 for unknown slugs
export const dynamic = 'force-static'; // Force static generation
```

**Configuration explained:**

- `dynamicParams = false`: Return 404 for slugs not in `generateStaticParams()` list instead of attempting dynamic rendering
- `dynamic = 'force-static'`: Force static generation; will error at build time if page uses dynamic APIs (helps catch issues)

### Results

Build output changed from:

```
ƒ /[locale]/blog/[slug]  (Dynamic)
```

to:

```
● /[locale]/blog/[slug]  (SSG)
  ├ /zh/blog/blog-iteration-journey
  ├ /zh/blog/building-my-blog
  └ [+10 more paths]
```

All blog posts are now pre-rendered as static HTML at build time, providing:

- **Faster page loads**: No server rendering needed
- **Better SEO**: Crawlers get complete HTML immediately
- **Lower costs**: Reduced server compute resources

## Issue 2: TableOfContents Memory Leak

### Symptoms

Browser console error:

```
TypeError: Cannot create property 'observer' on number '136'
```

### Root Cause

Looking at `components/TableOfContents.tsx`:

```typescript
useEffect(() => {
  const timer = setTimeout(() => {
    // ... create observer
    const observer = new IntersectionObserver(/* ... */);

    // ❌ Error: trying to create property on a number
    (timer as unknown as { observer: IntersectionObserver }).observer = observer;
  }, 150);

  return () => {
    clearTimeout(timer);
    // ❌ Observer not cleaned up
  };
}, []);
```

Issues:

1. `setTimeout` returns a number (timer ID)
2. Attempting to create `observer` property on a number throws error
3. Cleanup function doesn't disconnect IntersectionObserver, causing memory leak

### Solution

Hoist `observer` to `useEffect` scope:

```typescript
useEffect(() => {
  let observer: IntersectionObserver | null = null;

  const timer = setTimeout(() => {
    const items = getHeadings();
    if (items.length < 2) return;

    setHeadings(items);

    // ✅ Assign to scope variable
    observer = new IntersectionObserver(
      (entries) => {
        for (const entry of entries) {
          if (entry.isIntersecting) {
            setActiveId(entry.target.id);
          }
        }
      },
      { rootMargin: "0px 0px -70% 0px", threshold: 0 }
    );

    const article = document.querySelector("article");
    article?.querySelectorAll("h2, h3").forEach((node) => observer?.observe(node));
  }, 150);

  return () => {
    clearTimeout(timer);
    // ✅ Properly clean up observer
    observer?.disconnect();
  };
}, []);
```

**Improvements:**

1. `observer` as `useEffect` scope variable, accessible in cleanup
2. Cleanup function properly calls `disconnect()` to stop all observations
3. Avoids creating properties on primitive types

## Issue 3: Styling and Theme Improvements

### Color Contrast Enhancement

Original colors had insufficient contrast, affecting readability. Adjusted design tokens:

```css
/* Before */
:root {
  --fg: #555555;        /* Light gray */
  --fg-deep: #222222;
  --fg-light: #888888;
}

.dark {
  --c-bg: #050505;      /* Near black */
  --fg: #bbbbbb;        /* Light gray */
}

/* After */
:root {
  --fg: #333333;        /* Darker, higher contrast */
  --fg-deep: #111111;
  --fg-light: #666666;
}

.dark {
  --c-bg: #000000;      /* Pure black background */
  --fg: #e0e0e0;        /* Brighter, higher contrast */
  --fg-deep: #eeeeee;
  --fg-light: #a8a8a8;
}
```

### Code Block Language Labels

Added language labels to code blocks for easier identification:

```css
.prose pre > code[data-language]::before {
  content: attr(data-language);
  position: absolute;
  top: 0;
  left: 0;
  padding: 0.15rem 0.6rem;
  font-size: 0.65rem;
  text-transform: uppercase;
  letter-spacing: 0.05em;
  color: var(--fg-light);
  opacity: 0.6;
  background: rgba(125, 125, 125, 0.08);
  border-bottom-right-radius: 0.375rem;
}
```

Result: Code blocks now show labels like `TYPESCRIPT`, `BASH` in the top-left corner.

### Giscus Comments Theme Sync

Previously, comments used `preferred_color_scheme` but didn't update when switching themes. Changed to monitor theme changes:

```typescript
export default function Comments() {
  const [theme, setTheme] = useState<"light" | "dark">("light");

  // Monitor theme changes
  useEffect(() => {
    const updateTheme = () => {
      const isDark = document.documentElement.classList.contains("dark");
      setTheme(isDark ? "dark" : "light");
    };

    updateTheme();

    const observer = new MutationObserver(updateTheme);
    observer.observe(document.documentElement, {
      attributes: true,
      attributeFilter: ["class"],
    });

    return () => observer.disconnect();
  }, []);

  useEffect(() => {
    // ... reload Giscus script
    script.setAttribute("data-theme", theme);
    // ...
  }, [locale, theme]); // ✅ Depend on theme

  return <div ref={ref} />;
}
```

Now when switching themes, the comments section immediately reloads with the new theme.

## Summary

This optimization round addressed three key issues:

1. **Performance**: Forced static generation, all blog posts pre-rendered as HTML
2. **Memory leak fix**: Properly cleaned up IntersectionObserver
3. **UX improvements**: Better color contrast, code block labels, theme synchronization

These seemingly small improvements accumulate to significantly enhance user experience. Performance optimization isn't just about chasing numbers — it's about making users feel the speed and smoothness.
