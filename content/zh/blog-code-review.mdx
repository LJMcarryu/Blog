## 为什么要做代码审查？

博客上线后跑了一段时间，功能基本稳定。但"能跑"和"跑得好"是两回事——趁着还没积累太多技术债，我决定对整个项目做一次全面审查，把隐患和优化空间都摸清楚。

审查维度覆盖六个方面：**性能、安全、可访问性、SEO、代码质量、国际化**。

---

## 性能问题

### 视频背景资源过大

首页的背景视频是最大的性能瓶颈：

- `home_bg_video_dark.mp4` — 20.7 MB
- `home_bg_video_light.mp4` — 10.8 MB

总计超过 31 MB，移动端加载体验很差。优化方向：

1. 用 FFmpeg 压缩并转为 WebM/VP9 格式，体积可减少 60-70%
2. 给 `<video>` 添加 `poster` 属性，首帧用静态图代替
3. 考虑移动端直接用静态图替代视频

### 组件中的定时器隐患

`CodeCopyButton.tsx` 用 `setTimeout(100ms)` 等待 DOM 渲染完成后再插入复制按钮。这在慢设备上可能失效，更稳健的做法是用 `MutationObserver` 监听 DOM 变化。

`TableOfContents.tsx` 同样有 150ms 的延迟初始化，可能导致 CLS（累积布局偏移）。

### 主题切换时视频重载

`ThemedVideo.tsx` 在主题切换时调用 `.load()` 重新加载视频源，会导致画面闪烁。更好的方案是同时渲染两个 `<video>`，用 CSS `display` 控制显隐。

## 安全问题

### OG 图片参数未校验

`app/og/route.tsx` 从 URL ���数中读取 `title` 直接渲染到 `ImageResponse`，没有做长度限制或内容过滤。虽然 OG 图片生成环境相对安全，但作为外部输入的入口，建议加上基本校验。

### MDX 解析错误被静默吞掉

`lib/posts.ts` 的 catch 块直接返回空对象，不打日志也不抛错：

```typescript
// 当前写法 — 出错时完全静默
catch {
  return { slug, title: "", date: "", description: "", tags: [], content: "" };
}
```

建议至少加一行 `console.warn`，否则 MDX 文件写错了根本发现不了。

## 可访问性问题

### Lightbox 缺少焦点管理

`Lightbox.tsx` 用 `div` 模拟了一个模态弹窗，但没有实现焦点陷阱（focus trap）。键盘用户按 Tab 可以跳出弹窗，操作到背后的页面元素。

推荐方案：改用原生 `<dialog>` 元素，或者引入 `focus-trap-react` 库。

### 复制按钮的 aria-label 不一致

`CodeCopyButton.tsx` 的按钮文本是 "Copy"，但 `aria-label` 是 "Copy code"——屏幕阅读器和视觉呈现不一致，建议统一。

## SEO 问题

### 元数据描述不区分语言

`layout.tsx` 的 metadata description 写的是 `"个人博客 | Personal Blog"`，中英文混在一起。搜索引擎会根据页面语言判断描述是否匹配，建议改为根据 locale 动态设置。

### RSS 语言标签无效

`feed.xml/route.ts` 中 RSS 的 `language` 字段设为 `"zh-en"`，这不是合法的 RFC 5646 语言标签。应该为每个语言生成独立的 feed，或者使用 `"zh"` 作为主语言。

### Sitemap 缺少更新频率

`sitemap.ts` 生成的站点地图没有 `changeFrequency` 和 `priority` 字段。虽然 Google 声称不太依赖这些字段，但其他搜索引擎（如 Bing）仍然参考。

### robots.ts 的 fallback 风险

`robots.ts` 和 `sitemap.ts` 都用 `https://example.com` 作为 `NEXT_PUBLIC_SITE_URL` 的默认值。如果部署时忘记设置环境变量，站点地图和 robots.txt 会指向错误的域名，直接影响 SEO 收录。

## 代码质量

### CSS 中 `!important` 泛滥

`globals.css` 中有大量 `!important`：

- 20 个 `nth-child` 动画规则（第 90-109 行）
- 代码块背景色（第 223、227 行）
- Shiki 主题色（第 252-256 行）
- 行内代码样式（第 317-324 行）

`!important` 越多，后续维护越难。建议通过调整选择器层级或使用 CSS Layers 来解决优先级问题。

### 硬编码颜色值

代码块内的文字颜色直接写死了 `#0a0a0a` 和 `#f0f0f0`，没有走 CSS 变量体系。和其他地方用 `var(--fg)` 的风格不统一。

### 分页逻辑冗余

`blog/page.tsx` 中 `Math.max(1, Math.ceil(...))` 是多余的——`Math.ceil()` 对正数本身就返回 ≥ 1 的值。

### About 页面数据硬编码

`about/page.tsx` 的 SKILLS 数组直接写在组件里，而项目、书籍等数据都抽到了 `data/` 目录。建议保持一致，移到 `data/skills.ts`。

## 国际化问题

### 部分文案未走翻译系统

以下位置使用了硬编码的中英文字符串，没有通过 `next-intl` 的翻译函数：

- `blog/[slug]/page.tsx` 第 182 行 — 上一篇/下一篇标签
- `blog/page.tsx` 第 85 行 — "全部" / "All" 标签筛选

这些应该统一放到 `messages/zh.json` 和 `messages/en.json` 中。

### 类型断言不够健壮

`i18n/request.ts` 中用 `as "zh" | "en"` 做类型断言，如果将来新增语言（比如日语），这里会悄悄出 bug。建议从 `routing.locales` 推导类型。

## 第二轮审查：深层问题

完成前五轮优化后，我又做了一次深度审查，发现了一些更隐蔽的问题。

### CSS 变量覆盖不彻底

虽然之前引入了 `--border` 和 `--bg-subtle` 两个设计令牌，但 `globals.css` 中仍有大量裸写的 `rgba(125, 125, 125, ...)` 值散落在滚动条、代码块边框、行内代码、目录、引用块等样式中。这些值语义相同但没有走变量体系，改一个地方要改十几处。

### Giscus 初始化主题错误

`Comments.tsx` 的初始化 effect 依赖了 `[locale, theme]`，但 `theme` 的初始值是 `"light"`（React state 默认值），而不是页面实际的主题。这意味着暗色模式下首次加载 Giscus 时，iframe 会先以亮色主题渲染，然后 postMessage 再切换——用户会看到一次闪烁。

### 可访问性遗漏

- 汉堡菜单按钮缺少 `aria-controls` 属性，屏幕阅读器无法关联按钮和菜单
- 标签筛选的激活状态只有视觉样式（`data-active`），没有 `aria-current` 语义标记
- Lightbox 的 `alt` 文本写死为 `"Enlarged preview"`，丢失了原图的描述信息
- 搜索框的 `focus:ring-2` 没有指定颜色，焦点环在某些浏览器下不可见

### 冗余代码

- `blog/[slug]/page.tsx` 同时设置了 `dynamicParams = false` 和 `dynamic = 'force-static'`，后者是多余的
- `Footer.tsx` 和首页对 mailto 链接设置 `target="_self"`，这是浏览器默认行为
- RSS feed 的 description 混用中英文 `"个人博客 | Personal Blog"`，不符合 RSS 规范

## 第三轮审查：遗漏与冗余

第六轮优化后再次审查，发现了几个之前遗漏的问题。

### 设计令牌遗漏

`SearchBar.tsx` 中仍有一处 `rgba(125, 125, 125, 0.05)` 没有走变量体系——设计令牌迁移时漏掉了搜索框的背景色。

### 分页链接 URL 异常

`blog/page.tsx` 的 `baseHref` 在没有筛选参数时返回 `"?"` 而非空字符串，导致分页链接出现多余的问号（如 `/blog?`）。虽然功能不受影响，但 URL 不够干净。

### Giscus 语言切换失效

`Comments.tsx` 用 `initialized` ref 防止重复初始化，但这也阻止了切换语言时重新创建 Giscus iframe。用户切换中英文后，评论区的界面语言不会更新。

### CSS 变量冗余声明

`.dark` 块中有 11 个 CSS 变量和 `:root` 完全相同（所有 rgba 系列令牌、`--star`、`--success`）。这些值在明暗模式下不变，重复声明只增加维护负担。

### RSS 描述语言不当

RSS feed 的 `<description>` 使用纯中文，但 feed 实际包含中英文两种语言的文章，描述语言应该更通用。

## 第四轮审查：架构与 i18n

### 重复的暗色模式监听逻辑

`ThemedVideo.tsx`、`Comments.tsx` 和 `ThemeToggle.tsx` 三个组件各自创建了 MutationObserver 监听 `document.documentElement` 的 class 变化。前两个的逻辑完全相同，应该提取为共享的 `useDarkMode` Hook。

### UI 组件交互文案硬编码英文

四个客户端组件的 aria-label 和按钮文案是硬编码英文，没有走 i18n：

- `ThemeToggle` — "Switch to light/dark mode"
- `BackToTop` — "Back to top"
- `CodeCopyButton` — "Copy" / "Copied!" / "Failed" / "Copy code"
- `LanguageSwitcher` — "Toggle language"

中文用户看到的是英文交互提示，体验不一致。

### 缺少 canonical 和 alternate URL

布局和文章页面都没有设置 `alternates` 元数据。搜索引擎可能把 `/zh/blog/xxx` 和 `/en/blog/xxx` 当作重复内容，影响 SEO 收录。

### TableOfContents 不必要的重建

MutationObserver 在每次 DOM 变化时都重新初始化 IntersectionObserver，包括 CodeCopyButton 插入按钮时。应该缓存 heading 列表，只在标题实际变化时才重建。

---

## 第五轮审查：i18n 死角

### 阅读时间硬编码

`lib/reading-time.ts` 直接返回 `"X 分钟阅读"` / `"X min read"` 格式化字符串，没有走翻译系统。工具函数不应该包含 UI 文案。

### 书单笔记只有中文

`data/books.ts` 的书评笔记是纯中文字符串，英文用户看到的是中文内容。应该像 `projects.ts` 一样用 `Record<"zh" | "en", string>` 做双语。

### Stars 组件 aria-label 硬编码

`books/page.tsx` 的星级评分 `aria-label` 写死了英文 `"X stars"`，中文用户的屏幕阅读器会读出英文。

### 照片 alt 文本 fallback 失效

`photos/page.tsx` 的 `altFromPath()` 用 `??` 做 fallback，但 `??` 不捕获空字符串。如果文件名只有扩展名（如 `.jpg`），alt 会是空的。

## 总结

整体来看，项目架构清晰，Next.js App Router + next-intl 的组合用得比较规范，暗色模式和动画系统也做得不错。经过五轮审查，主要的优化空间集中在五个方面：

1. **性能**：视频资源压缩是最大的收益点
2. **健壮性**：错误处理和环境变量校验需要补齐
3. **一致性**：CSS 变量体系需要彻底贯穿，不能留死角
4. **可访问性**：aria 属性、焦点管理、语义标记、交互文案 i18n 需要逐一补齐
5. **SEO**：canonical/alternate URL、结构化数据需要完善

这些问题没有一个是"紧急"的，但逐步修复会让项目的可维护性上一个台阶。下一步我会按优先级逐个解决。
