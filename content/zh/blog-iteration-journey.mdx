## 第一阶段：代码审计（第 1-4 轮）

### 发现的典型问题

搭完博客后我对整个代码库做了一次全面审计，发现了 33 个问题。举几个有代表性的：

**空日期导致排序 NaN：**

```typescript
// ❌ 之前：空日期会产生 NaN，排序结果不确定
.sort((a, b) => new Date(b.date) - new Date(a.date))

// ✅ 之后：空日期回退为时间戳 0
.sort((a, b) => {
  const ta = a.date ? new Date(a.date).getTime() : 0;
  const tb = b.date ? new Date(b.date).getTime() : 0;
  return tb - ta;
});
```

**OG 描述含 MDX 标记：**

社交平台预览显示的描述里混着 `#`、`*`、`import` 语句。修复方法是先用正则清洗，再截取前 160 字符。

**数据硬编码在组件中：**

项目列表、书单、工具链等数据全写在页面组件里，改一个数据要翻一堆 JSX。提取到 `data/` 目录后，数据和视图彻底分离。

### 审计的方法论

我的审计清单覆盖了这些维度：

- **安全**：XSS、外部链接 `rel`、硬编码密钥
- **TypeScript**：严格模式、类型断言、未使用代码
- **React**：key 稳定性、Server/Client 划分、useEffect 清理
- **无障碍**：aria 标签、语义化 HTML、键盘导航
- **SEO**：metadata、sitemap、robots.txt、OG 标签
- **i18n**：消息键一致性、locale 覆盖

经过 4 轮迭代，代码库达到了 **ESLint 零警告、TypeScript 零错误** 的状态。

## 第二阶段：功能分层开发（第 5-7 轮）

修完所有 bug 后，我把想要的功能分成三个梯队，按优先级逐步实现。

### 第一梯队：核心体验

这些是"没有就显得不专业"的功能：

- **代码语法高亮**：`rehype-pretty-code` + Shiki，支持亮/暗双主题
- **RSS Feed**：RSS 2.0 格式，遍历所有 locale 的文章
- **阅读时间**：中文按 400 字/分钟，英文按 200 词/分钟
- **Favicon**：用 `next/og` 动态生成，不用手动切图
- **移动端导航**：640px 断点，汉堡菜单 + 滑入动画

### 第二梯队：用户体验

这些是"有了会让用户多待一会"的功能：

- **标签系统**：frontmatter 加 `tags` 字段，支持 URL 参数过滤
- **目录组件**：IntersectionObserver 实时高亮当前章节
- **代码复制按钮**：悬停显示，点击后 "Copied!" 反馈
- **上一篇/下一篇**：文章底部双向导航
- **返回顶部**：滚动 400px 后浮现
- **图片灯箱**：点击放大，Escape 关闭
- **网站 Footer**：社交链接 + 版权信息

### 第三梯队：生产就绪

这些是"上线前的最后一公里"：

- **OG 图片生成**：Edge Runtime 动态生成 1200×630 分享图
- **JSON-LD**：文章页 Article schema + 全站 WebSite schema
- **Umami 分析**：环境变量控制，不设置则不加载
- **博客分页**：每页 10 篇，兼容搜索和标签过滤
- **单元测试**：Vitest 覆盖核心库函数，17 项测试全部通过

## 分层策略的好处

把功能分成三个梯队而不是一次性全做，有几个好处：

1. **每轮都能 build 通过**——不会出现"改了一半跑不起来"
2. **优先级清晰**——先做影响最大的，而不是最有趣的
3. **每轮都是可交付状态**——任何时候都可以部署
