## 前言

在[上一篇进阶指南](/zh/blog/claude-code-advanced-guide)里，我简单提到了 CLAUDE.md 是 Claude Code 的"第一个效率杠杆"。但很多人看完还是不知道该怎么写——写什么、写多少、怎么组织。

这篇文章就专门把 CLAUDE.md 讲透。

---

## 一、CLAUDE.md 是什么

一句话：**CLAUDE.md 是你给 Claude Code 写的项目说明书。**

每次启动 Claude Code，它会自动读取 CLAUDE.md 并注入到系统提示词中。这意味着你不需要每次对话都重复"我用的是 Next.js 16""提交信息用中文"这些背景信息——写一次，永久生效。

你可以把它理解为：
- 给 AI 的 onboarding 文档
- 项目的"宪法"
- 一份可执行的编码规范

**和 README 的区别：** README 是给人看的，CLAUDE.md 是给 AI 看的。README 可以写得优美详尽，CLAUDE.md 要写得精准可执行。

## 二、配置层级

CLAUDE.md 有三个层级，Claude Code 会全部加载并合并：

```
~/.claude/CLAUDE.md              → 全局（所有项目生效）
项目根目录/CLAUDE.md              → 项目级（提交到 Git，团队共享）
子目录/CLAUDE.md                  → 目录级（模块专属规则）
```

### 2.1 全局 CLAUDE.md

放你的个人偏好，适用于所有项目：

```markdown
## 通用偏好
- 回复使用中文
- Git 提交信息用中文，格式：类型: 描述
- 优先使用 TypeScript
- 代码注释用英文
- 不要自动提交代码，等我确认
```

路径：`~/.claude/CLAUDE.md`

### 2.2 项目级 CLAUDE.md

放项目的技术栈、架构和规范。这是最重要的一层，建议提交到 Git 让团队共享：

```markdown
# 我的博客项目

## 技术栈
- Next.js 16 (App Router) + TypeScript
- Tailwind CSS v4
- next-intl (zh/en 双语)

## 常用命令
- npm run dev — 启动开发服务器
- npm run build — 构建
- npm test — 跑测试
```

路径：项目根目录 `/CLAUDE.md`

### 2.3 目录级 CLAUDE.md

在 monorepo 或大型项目中特别有用。比如给组件目录单独写���范：

```markdown
# components/

## 组件规范
- 每个组件一个文件，PascalCase 命名
- 必须导出 Props 类型
- 客户端组件必须加 'use client' 指令
- 优先用 server component
```

路径：`components/CLAUDE.md`

### 2.4 优先级

当多层配置冲突时，**越具体的层级优先级越高**：

```
目录级 > 项目级 > 全局
```

## 三、写什么、不写什么

### 该写的

核心原则：**只写 Claude 无法从代码中推断出来的信息。**

| 该写 | 为什么 |
|------|--------|
| 技术栈和版本 | Claude 不一定能准确判断你用的版本 |
| 构建/测试/部署命令 | 每个项目不一样 |
| 代码风格约定 | 和默认不同的规则才需要写 |
| 架构决策和边界 | "不要修改 middleware.ts" |
| 项目特有术语 | 业务领域的概念 |
| 文件结构说明 | 特别是非常规的目录组织 |
| 禁止事项 | 明确的红线 |

### 不该写的

| 不该写 | 为什么 |
|--------|--------|
| Claude 能从代码读到的信息 | 浪费 token |
| README 的内容复制 | 重复信息 |
| 过于详细的 API 文档 | 太长会稀释重点 |
| 显而易见的规则 | "变量名要有意义"——这不需要说 |

## 四、编写技巧

### 4.1 具体、可执行

差的写法：
```markdown
写好代码，注意质量
```

好的写法：
```markdown
- 函数不超过 30 行
- 组件 Props 必须定义 TypeScript 接口
- 所有 API 调用必须有错误处理
```

### 4.2 用祈使句

CLAUDE.md 是指令，不是描述。用"做什么"而不是"我们通常会做什么"：

```markdown
# ✗ 不好
我们的项目通常使用 ESM 模块格式

# ✓ 好
使用 ESM 模块格式，不要用 CommonJS
```

### 4.3 分区清晰

用 Markdown 标题分区，把最重要的信息放在前面：

```markdown
## 技术栈        ← 最先看到
## 常用命令      ← 高频使用
## 代码规范      ← 日常约束
## 架构说明      ← 需要时参考
## 禁止事项      ← 红线
```

### 4.4 保持精简

CLAUDE.md 会占用上下文窗口。写得太长，反而会稀释重要信息。经验法则：**控制在 100-200 行以内。**

如果确实需要更多内容，可以拆分到子目录的 CLAUDE.md 中——Claude 只在进入该目录时才会加载。

### 4.5 持续迭代

CLAUDE.md 不是写一次就完事的。像对待代码一样对待它：

- 发现 Claude 反复犯同一个错？加一条规则
- 某条规则从没生效过？删掉它
- 项目技术栈升级了？同步更新
- Code Review 时也 review CLAUDE.md

## 五、CLAUDE.md vs settings.json

很多人搞混这两个文件，它们的职责完全不同：

| | CLAUDE.md | .claude/settings.json |
|---|-----------|----------------------|
| 格式 | Markdown | JSON |
| 作用 | 项目指令和上下文 | 工具配置和权限 |
| 内容 | 技术栈、规范、架构 | Hooks、MCP、权限模式 |
| 共享 | 提交到 Git，团队共享 | 通常 gitignore |
| 类比 | 员工手册 | 系统设置 |

简单说：**CLAUDE.md 告诉 Claude "做什么"，settings.json 告诉 Claude "怎么做"。**

## 六、实战示例一：Next.js 博客项目

这就是我这个博客项目实际使用的 CLAUDE.md（精简版）：

```markdown
# 个人博客

## 技术栈
- Next.js 16 (App Router) + TypeScript 严格模式
- Tailwind CSS v4（无 tailwind.config.js，用 CSS @import "tailwindcss"）
- next-intl 做 i18n，支持 zh/en，默认 zh
- MDX 内容通过 next-mdx-remote/rsc 渲染
- rehype-pretty-code 做代码高亮（双主题：github-light / github-dark-dimmed）

## 常用命令
- npm run dev — 启动开发服务器（不要帮我运行）
- npm run build — 构建生产版本
- npm test — 运行 vitest 测试
- npx eslint . — 检查代码规范

## 项目结构
- app/[locale]/ — 基于 locale 的路由
- content/zh/*.mdx, content/en/*.mdx — 博客内容（中英双语）
- components/ — React 组件
- lib/posts.ts — 博客文章加载逻辑
- i18n/routing.ts — 国际化路由配置，导出 Locale 类型
- messages/zh.json, en.json — UI 翻译字符串

## 代码规范
- 组件用 PascalCase，工具函数用 camelCase
- 优先用 Server Component，只在需要交互时加 'use client'
- locale 参数类型用 Locale（从 i18n/routing.ts 导入），不要用 string
- CSS 变量定义在 globals.css，不要硬编码颜色值
- 暗色模式用 class-based（.dark），通过 CSS 变量切换

## 国际化
- 所有用户可见文本必须走 next-intl 的 useTranslations
- 翻译 key 在 messages/zh.json 和 messages/en.json 中同步维护
- 新增 UI 文本时，两个语言文件都要更新

## Git 提交
- 提交信息用英文，格式：type: description
- type 包括：feat, fix, refactor, chore, docs, test, style
- 不要自动 push，等我确认

## 禁止事项
- 不要修改 middleware.ts（next-intl 中间件配置）
- 不要在 Server Component 中使用 useState/useEffect
- 不要引入新的 CSS 框架或 UI 库
- 不要删除或修改现有的 CSS 变量
```

### 为什么这样写

1. **技术栈写了版本号** — "Tailwind CSS v4" 和 v3 的配置方式完全不同，不写版本 Claude 可能用错误的方式
2. **明确了 Tailwind v4 的特殊性** — 没有 tailwind.config.js，用 CSS @import，这是 v4 独有的
3. **Locale 类型约束** — 不写这条，Claude 会用 `string` 类型，导致类型不安全
4. **禁止事项很具体** — 不是"注意不要破坏东西"，而是明确列出哪些文件不能动

## 七、实战示例二：Flutter 项目

假设你在开发一个 Flutter 电商 App，CLAUDE.md 可以这样写：

```markdown
# ShopEasy — Flutter 电商 App

## 技术栈
- Flutter 3.27 + Dart 3.6
- 状态管理：Riverpod 2.x（不要用 Provider 或 Bloc）
- 路由：go_router 14.x（声明式路由）
- 网络请求：dio 5.x + retrofit
- 本地存储：shared_preferences（轻量）+ drift（数据库）
- 国际化：flutter_localizations + intl

## 常用命令
- flutter run — 运行 App（不要帮我执行）
- flutter test — 运行单元测试
- flutter analyze — 静态分析
- dart run build_runner build --delete-conflicting-outputs — 代码生成
- flutter gen-l10n — 生成国际化文件

## 项目结构（Feature-First）
- lib/
  - core/ — 通用工具、主题、常量、网络层
    - theme/ — AppTheme，颜色和字体定义
    - network/ — Dio 配置、拦截器、API 基类
    - constants/ — API 端点、存储 key
  - features/ — 按功能模块组织
    - auth/ — 登录注册
      - data/ — Repository 实现、数据源、DTO
      - domain/ — Entity、Repository 接口
      - presentation/ — 页面、Widget、Controller
    - home/ — 首页
    - cart/ — 购物车
    - product/ — 商品详情
    - profile/ — 个人中心
  - shared/ — 跨模块共享的 Widget 和工具
  - app.dart — MaterialApp.router 入口
  - main.dart — 初始化和启动

## 架构规范
- 严格遵循 Clean Architecture 三层分离：data → domain → presentation
- domain 层不依赖任何外部包（纯 Dart）
- Repository 模式：domain 定义接口，data 层实现
- 状态管理统一用 Riverpod：
  - 页面状态用 AsyncNotifierProvider
  - 简单状态用 StateProvider
  - 依赖注入用 Provider
- 所有 API 返回值用 Result<T> 类型包装（lib/core/network/result.dart）

## 代码规范
- 文件名用 snake_case，类名用 PascalCase
- Widget 拆分粒度：超过 80 行的 build 方法必须拆分子 Widget
- 私有方法以 _ 开头
- 所有公开 API 必须写 dartdoc 注释
- 使用 const 构造函数优化 Widget 重建
- 禁止在 Widget 中直接调用 API，必须通过 Repository

## 测试
- 单元测试覆盖所有 Repository 和 Notifier
- Widget 测试覆盖关键交互流程
- Mock 用 mocktail（不要用 mockito）
- 测试文件和源文件同名，放在 test/ 对应目录下

## 禁止事项
- 不要用 setState（统一用 Riverpod）
- 不要在 presentation 层直接导入 data 层
- 不要硬编码颜色值，必须用 Theme.of(context)
- 不要用 print 调试，用 logger 包
- 不要修改 core/network/dio_client.dart 的拦截器链
```

### 为什么这样写

1. **明确了状态管理方案** — Flutter 生态有太多状态管理方案，不写清楚 Claude 可能用 setState 或 Provider
2. **Feature-First 结构详细列出** — 这不是 Flutter 默认结构，Claude 需要知道文件该放哪
3. **Clean Architecture 边界** — "domain 层不依赖外部包"这条规则，不写的话 Claude 很容易违反
4. **具体到包名和版本** — "mocktail 不要用 mockito"，这种细节决定了生成代码能不能直接跑
5. **Widget 拆分规则量化** — "超过 80 行必须拆分"比"保持 Widget 简洁"有用得多

## 八、进阶技巧

### 8.1 用 /init 自动生成

如果你不想从零开始写，Claude Code 提供了 `/init` 命令：

```bash
claude /init
```

Claude 会分析你的项目结构、package.json、配置��件等，自动生成一份 CLAUDE.md 初稿。你在这个基础上修改，比从空白开始高效得多。

### 8.2 团队协作

把项目级 CLAUDE.md 提交到 Git：

```bash
git add CLAUDE.md
git commit -m "chore: add CLAUDE.md for AI-assisted development"
```

这样团队每个人用 Claude Code 时都会加载同一份规范，保证 AI 生成的代码风格一致。

Code Review 时也应该 review CLAUDE.md 的改动——它和 ESLint 配置一样重要。

### 8.3 Monorepo 策略

在 monorepo 中，善用目录级 CLAUDE.md：

```
monorepo/
  CLAUDE.md                    ← 全局：通用规范、Git 约定
  packages/
    frontend/
      CLAUDE.md                ← 前端：React 规范、组件约定
    backend/
      CLAUDE.md                ← 后端：API 规范、数据库约定
    shared/
      CLAUDE.md                ← 共享：类型定义、工具函数规范
```

Claude 进入某个子目录工作时，会同时加载根目录和该子目录的 CLAUDE.md。

### 8.4 配合 settings.json

CLAUDE.md 管"做什么"，settings.json 管"怎么做"。配合使用效果最好：

```
CLAUDE.md          → 保存后用 Prettier 格式化
settings.json      → 配置 PostToolUse Hook 自动执行 Prettier
```

```
CLAUDE.md          → 提交前必须通过 ESLint
settings.json      → 配置 PreToolUse Hook 拦截不合规的提交
```

## 九、常见问题

**Q: CLAUDE.md 太长会怎样？**
A: 会占用上下文窗口，挤压实际对话的空间。建议控制在 200 行以内，把详细内容拆到子目录。

**Q: CLAUDE.md 和 Cursor Rules / Copilot Instructions 是什么关系？**
A: 思路一样，都是给 AI 的项目级指令。但格式和加载机制不同，不能互通。如果你同时用多个 AI 工具，可能需要维护多份类似的文件。

**Q: 写了 CLAUDE.md 但 Claude 好像没遵守？**
A: 几个可能的原因：
1. 规则太模糊——"写好代码"不如"函数不超过 30 行"
2. 规则互相矛盾——检查是否有冲突的指令
3. 上下文太长——CLAUDE.md 被压缩了，把关键规则放在文件开头
4. 对话中的指令覆盖了——对话中的明确指令优先级高于 CLAUDE.md

**Q: 要不要把 CLAUDE.md 加到 .gitignore？**
A: 看情况。项目级的建议提交（团队共享），但如果包含个人偏好或敏感路径，可以用 `~/.claude/CLAUDE.md` 放全局配置，不提交。

## 总结

CLAUDE.md 的本质就是一份**给 AI 的工作手册**。写好它，你和 Claude Code 的协作效率会有质的提升。

记住三个原则：
1. **只写 Claude 推断不出来的信息**
2. **具体、可执行、有约束**
3. **持续迭代，像对待代码一样对待它**

花 30 分钟写一份好的 CLAUDE.md，能帮你在未来的每次对话中节省大量时间。这可能是 ROI 最高的 30 分钟。
